# ChromaDB 1.5.8 segfault race in the nightly integrity check

- **Incident date:** 2026-05-13 (first rollback-triggering crash 01:31 local; crash cluster spans 2026-05-12 04:01 → 2026-05-13 04:01, UTC+2)
- **Resolved in:** TAILOR v1.2.9 (2026-05-14)
- **Affected component:** nightly integrity check in [sync_and_ingest.sh](../../sync_and_ingest.sh)
- **Upstream component:** `chromadb` 1.5.8 (`chromadb_rust_bindings.abi3.so`)
- **Status:** resolved; one residual structural risk tracked (see §9)

## Summary

Five `SIGSEGV` in `chromadb_rust_bindings.abi3.so` over 48 hours triggered one spurious auto-rollback on a healthy database. The crash is a deterministic race, not a flake: any second `PersistentClient` opening a chromadb 1.5.8 directory while a live client holds file descriptors on it crashes with `KERN_INVALID_ADDRESS @ 0x0`. The integrity-check wrapper compounded the upstream defect by reading captured stdout but not the subprocess exit code, so a signal-killed Python (empty output) was indistinguishable from a deterministic corruption report — and the rollback fired. Three fixes shipped on the same day: signal-aware retry (defense-in-depth), an APFS clone-copy of `chroma.sqlite3` so the check runs against an isolated snapshot (structural), and a full-DB tar.gz backup that closes a separate, latent rollback-target gap.

## 3. Impact

One auto-rollback fired on a healthy database at 2026-05-13 01:31. No user-visible data loss is confirmed from the available material — but the rollback path was a *latent* HNSW-desync trap regardless of whether desync materialized this time: the artifact it restored from (`chroma_*.sqlite3.gz`) contains only `chroma.sqlite3` and not the HNSW segment directories under `db/<uuid>/`. Restoring `chroma.sqlite3` to a state from N hours ago, against vector segment files mutated since then, leaves segment-metadata pointers stale.

The deterministic-corruption safety net (`EMPTY:0` and `ERROR:<msg>` branches) was not invalidated by this incident — only the signal-killed branch was. Integrity-check + rollback as a concept remained sound; the trigger predicate did not.

## 4. Root cause

A structural race between two clients of the same chromadb 1.5.8 directory:

1. **Live client.** The MCP server opens a `PersistentClient` at daemon start and holds it for the process lifetime. File descriptors on `db/chroma.sqlite3` and the HNSW segment files remain open continuously.
2. **Check client.** The nightly integrity check spawned a Python subprocess from [sync_and_ingest.sh](../../sync_and_ingest.sh) that opened its own `PersistentClient` against the same directory, called `col.count()`, and exited.

Under chromadb 1.5.8, the second client crashes with `SIGSEGV` in the Rust binding (`KERN_INVALID_ADDRESS @ 0x0`) when the first client holds the relevant FDs open. P0bis confirmed the determinism: in the actual deployment configuration, the race is not probabilistic but contractual — the second client *will* crash, every time, until the first releases.

The exact upstream defect is not characterized in the available material. The deployment-side observation is sufficient to act on: chromadb 1.5.8 does not safely support a second `PersistentClient` against a directory whose files are held by a live client.

## 5. Why the integrity-check wrapper failed-open in the wrong direction

The bash wrapper looked roughly like this (paraphrased from `93a4790`'s pre-state):

```bash
CHROMA_CHECK=$(.venv/bin/python3 - <<'PY'
# open PersistentClient, col.count(),
# print "OK:<n>" / "EMPTY:0" / "ERROR:<msg>"
PY
)

if [[ "$CHROMA_CHECK" != OK:* ]]; then
    # trigger auto-rollback from chroma_*.sqlite3.gz
fi
```

A subprocess exposes two information channels: standard output and the exit code. The Python script was designed to emit exactly one of three lines on stdout (`OK:<n>`, `EMPTY:0`, `ERROR:<msg>`) — the *content* of stdout was the signal, not the exit code. That is a reasonable design for a script that always runs to completion.

When the subprocess is killed by a signal, it does neither: stdout is empty (no buffered line was flushed before the crash) and the exit code is `128 + signo` (139 for `SIGSEGV`). The wrapper captured stdout via `$(…)` and never inspected `$?`. An empty `CHROMA_CHECK` failed the `OK:*` glob in exactly the same way as a deterministic `ERROR:…`, and the rollback branch fired.

The category error: treating "the check could not run" the same as "the check ran and reports corruption". These have the same *string* representation (anything not starting with `OK:`) but they are genuinely different states with different correct responses:

- *The check ran and reports corruption.* Live DB is bad. Restore from backup is the right action.
- *The check could not run.* Live DB state is unknown. Restoring from backup destroys what may be the only good copy and replaces it with an older state.

A wrapper that decides whether to **destroy and restore data** must distinguish these. The 01:31 rollback ran because it did not.

## 6. Fixes deployed

Three commits on 2026-05-13, in landing order.

### P1 — signal-aware retry ([`93a4790`](https://github.com/tailormemory/tailor/commit/93a4790))

Three attempts total (one initial + two retries), 5 s sleep between, retry gated on `exit > 128`: only signal-killed Python (`SIGSEGV` = 139, `SIGABRT` = 134, `SIGKILL` = 137) is retried. Deterministic exits (`EMPTY:0`, `ERROR:<msg>`, Python exit 1) still trigger immediate rollback — that path is the genuine-corruption safety net and must remain fast. The Python source is hoisted to a heredoc variable so retries don't re-expand it and `$?` is read cleanly outside the `$(...)` capture. Failure logs now include `rc`, `status`, and raw output rather than the bare empty `$CHROMA_CHECK` that was useless during the incident.

Worst-case added latency if the first two attempts both segfault: ~10 s on a 75–90 min pipeline.

### G — COW snapshot ([`2fefc16`](https://github.com/tailormemory/tailor/commit/2fefc16))

P0bis confirmed the segfault is deterministic; P1 alone would have failed all three retries the next time the race fired. G snapshots `db/chroma.sqlite3` into `/tmp/chroma_check_$$` using APFS clone-copy (≈ 1.5 s for the 2.5 GB live file) and runs the check against the isolated directory. No second `PersistentClient` ever touches the live path; the race is structurally eliminated.

Implementation details that matter:

- Only `chroma.sqlite3` is cloned. `col.count()` reads the SQLite metadata segment and does not need the HNSW segment files. `PersistentClient` takes a directory path, not a file, so the snapshot is a directory containing the cloned SQLite.
- Snapshot path is PID-based (`/tmp/chroma_check_$$`) to survive stale dirs from prior runs. Cleanup via an idempotent `EXIT` trap plus an explicit `rm -rf` after success.
- `cp` failure (disk full, permissions) triggers a distinct alert and exits 1 **without** rollback. The live DB is fine in that case; only the check infrastructure broke. This is the explicit "could not run" branch that the pre-incident wrapper lacked.

P1 is retained on top of G as defense-in-depth (see §7).

### P3 — full-DB tar.gz backup ([`eb506d2`](https://github.com/tailormemory/tailor/commit/eb506d2))

P0bis exposed a separate gap. `chroma_*.sqlite3.gz` — the artifact consumed by the auto-rollback path at `sync_and_ingest.sh:321` — backs up only `chroma.sqlite3`, leaving HNSW segment dirs, `facts.sqlite3`, `entity_index.sqlite3`, and other runtime state outside the snapshot. A rollback restores `chroma.sqlite3` to a coherent point in the past while on-disk vector segments and facts data have moved forward — the classic HNSW desync trap.

P3 adds `tailor_db_full_<TS>.tar.gz` on the same 07:30 schedule with an explicit whitelist: `chroma.sqlite3` + HNSW UUID dirs (discovered via UUID-v4 regex), `facts.sqlite3`, `entity_index.sqlite3`, `doc_registry.json`, `reminders.json`, user-profile / translation / supplement JSON files, and the encrypted `secrets.sqlite3` blob (decrypt key lives outside the backup). `KEEP=3`. Smoke test 2026-05-13 20:05: 2 m 18 s wall, 1.85 GB tar.gz, `gzip -t` OK, all five HNSW segment files present.

P3 is currently a manual-restore artifact. The fast-rollback callsite at `sync_and_ingest.sh:321` was deliberately left unchanged for v1.2.9 — see §9.

## 7. Why three fixes and not one

Each layer addresses a failure class the others cannot catch.

**G is the structural fix.** Snapshotting `chroma.sqlite3` and running the check against the clone eliminates the live-path race entirely. If G alone were trustworthy, P1 and P3 would be unnecessary for this incident.

**G alone is not sufficient.** G introduces two new failure modes that did not exist before: `cp` failure (disk full, permissions, transient filesystem error) and `chromadb` crashes against the *isolated* snapshot. The first is handled with the no-rollback exit path described above. The second is the interesting one: a segfault on an isolated snapshot would indicate a *fresh bug class* — not the FD-sharing race that G addresses, but something else (corrupted SQLite, a new chromadb regression, ENOSPC in `/tmp` mid-check). The wrong response to that signal is to silently re-run forever. The right response is to surface it.

**P1 is retained because it surfaces the signal cheaply.** Three attempts with signal-aware retry is the smallest amount of code that distinguishes "transient" (retry succeeds, log a note) from "persistent" (all retries fail, rollback fires with a useful log line including `rc`, `status`, and raw output). Without P1, a segfault on an isolated snapshot would either be retried indefinitely or treated as immediate "no answer = corruption" — the original failure mode, now even less informative because the snapshot makes a crash unexpected.

**P3 is orthogonal.** The HNSW desync gap in `chroma_*.sqlite3.gz` would have existed even if the segfault had never happened. P3 closes it independently. The full-DB tar.gz is the artifact a future "we genuinely need to restore" scenario should consume; that callsite migration is tracked in §9.

The defense-in-depth here is not three copies of the same fix — it is three layers each justified by what the other two cannot catch. That distinction matters: stacking redundant fixes accumulates complexity without buying robustness. Stacking complementary fixes does.

## 8. Generalizable lessons

These apply outside TAILOR — to anyone running chromadb 1.5.x, or more generally to anyone running scheduled "is the DB healthy?" checks against a live embedded database with auto-remediation on failure.

**Exit code and stdout serve different purposes; capture and branch on the right one.** A shell wrapper that captures stdout via `$(…)` and discards `$?` is committing to "the program will tell me what happened in its output". That contract holds when the program runs to completion. It breaks the instant the program is killed by a signal, hits a Python interpreter assertion, or panics without flushing. Wrappers whose decision tree includes destroy-and-restore must read `$?` and branch on it first, before parsing output.

**"Read-only" checks against an embedded DB are not OS-level read-only.** They open file descriptors, acquire SQLite shared locks, may `mmap` segment files. Whether that interferes with a live client depends on the engine — SQLite generally tolerates concurrent readers; chromadb 1.5.8's Rust binding, as observed here, does not. Treat any second client as potential interference, not a guaranteed bystander, and verify against the specific engine version in use.

**Snapshot-then-check is structurally safer than retry-on-flake when the filesystem supports cheap COW.** APFS clone, btrfs reflink, and ZFS clone all produce a snapshot in O(metadata), not O(file size). A 2.5 GB clone in 1.5 seconds is cheap enough to do on every nightly run. Retries are a probabilistic mitigation — they reduce the chance of the failure firing but do not eliminate the failure mode. Snapshots eliminate it: the live-path race cannot fire if no client opens the live path.

**Auto-rollback predicates must distinguish "check could not run" from "check ran and found corruption".** These have different correct responses. The first warrants alerting an operator and leaving state untouched. The second warrants restore-from-backup. A predicate that collapses both into a single "not OK" branch will eventually restore over a state that was fine — the inverse of the safety property auto-rollback exists to provide.

**A "database" that is actually multiple co-dependent files needs a multi-file backup, or it is a desync trap.** chromadb stores SQLite metadata in one file and vector segments in a separate directory; their states must be consistent. Backing up only the SQLite file produces a snapshot that, when restored against a moved-forward segment directory, yields a database in a state that never existed. The same applies to any storage layout with separate index and main-data files, separate WAL and main files where the WAL is not checkpointed, or split read/write paths. Either back up all co-dependent files atomically, or back up nothing and document the rebuild-from-source procedure.

**Defense-in-depth is not the same as redundant fixes.** Each additional layer must be justified by a failure class the existing layers cannot catch. If a second fix is a smaller version of the first, it adds maintenance cost without adding robustness. If a second fix addresses a different failure class — as P1 addresses "fresh bug class" while G addresses "FD-sharing race" — it earns its place.

## 9. Open questions and residual risks

These are inferred from the available commit bodies and CHANGELOG entries. They are not a complete roadmap.

**Migrate the fast-rollback callsite to consume the full-DB artifact.** `sync_and_ingest.sh:321` still restores from `chroma_*.sqlite3.gz`, the partial artifact. Until that callsite is migrated to `tailor_db_full_*.tar.gz`, the HNSW desync gap described in §3 remains live for any future rollback. The full-DB artifact is currently a manual-restore-only path. Tracked in [`eb506d2`](https://github.com/tailormemory/tailor/commit/eb506d2)'s body as "a separate task".

**Should chromadb be pinned to a non-1.5.8 version?** Open question. Three distinct chromadb 1.5.8 incidents have shipped fixes in nine days — [§ 1.2.6.3](https://github.com/tailormemory/tailor/blob/v1.2.9/CHANGELOG.md#1263--2026-05-04--chromadb-158-silent-drop-surface-resolve-recover) (2026-05-04, silent drop), [§ 1.2.7](https://github.com/tailormemory/tailor/blob/v1.2.9/CHANGELOG.md#127--2026-05-06--operational-hardening-for-chromadb-158-frozen-state-recovery) (2026-05-06, frozen-state recovery), [§ 1.2.9](https://github.com/tailormemory/tailor/blob/v1.2.9/CHANGELOG.md#129--2026-05-14--scheduler-resilience--nightly-hardening) (this post-mortem). That cadence is enough to ask whether 1.5.7 or 1.6.x would be a better baseline, but the material on hand does not establish whether 1.5.7 was unaffected by the same race or whether 1.6.x ships a fix. A controlled comparison has not been done.

**Other code paths exhibit the same structural race shape.** Any subprocess that opens a `PersistentClient` against the live chromadb directory while MCP is up is subject to the same crash — `ingest_gmail.py` is one example. The mitigation in G is local to the integrity check in [sync_and_ingest.sh](../../sync_and_ingest.sh); it does not protect ingestion or extraction paths. Two options for closing this systemically: extend the COW-snapshot pattern into a shared helper that any subprocess can opt into, or route all chromadb access through MCP and forbid subprocess clients entirely. Neither is implemented as of v1.2.9.

**Has the upstream race been reported to chromadb?** Not established in the available material. Anyone hitting the same crash signature (`KERN_INVALID_ADDRESS @ 0x0` in `chromadb_rust_bindings.abi3.so`, second `PersistentClient` against a directory held by a live client) is encouraged to check the upstream issue tracker before re-debugging it.

## 11. Errata — 2026-05-16: the v1.2.6.3 silent-drop fix covered 4 of 7 chromadb write callsites

**Added 2026-05-16.** This errata does not revise §§1–10 above; it documents what was discovered three days after publication.

### What this post-mortem implied

§9 cites v1.2.6.3 in the "shipped fixes" cadence under the CHANGELOG label *silent-drop-surface-resolve-recover*, and §10 links the same entry. Read together, the implication is that the silent-drop class was closed by the `verified_upsert()` introduced in v1.2.6.3. The scope of the fix was narrower than that.

### What was found

The 3-ghost cluster between the 2026-05-07 audit (count = 0) and the 2026-05-16 audit (count = 3) triggered a callsite-by-callsite walk of every direct `collection.add(...)` and `collection.upsert(...)` in the codebase. `verified_upsert()` was adopted by the 4 batch ingest scripts that the v1.2.6.3 commit explicitly wired — `ingest_upload.py`, `ingest_conversations.py`, `ingest_docs.py`, `ingest_gmail.py` — and **not** by the 3 mcp_server-side write paths: `kb_add()`, `kb_update_session()`, and the `/api/ingest-live` HTTP endpoint. The three confirmed ghosts matched the three missed callsites: one `claude_<ts>_<hash>` from `kb_add` on the evening of 2026-05-12, and two `live_claude_<hex16>` from `/api/ingest-live`.

The walk also exposed a co-discovery: `tests/test_ingest_helpers.py` had been 10/12 red on `main` since v1.2.6.3 introduced the sample-get sentinel without updating the `FakeCollection` mock with a `.get()` method. The regression did not surface for three weeks because the suite was not run. The v1.2.6.3 fix was not, in any meaningful sense, verified.

The "invisible class" — drops that leave neither SQL nor HNSW trace — was the design target of `verified_upsert()`'s sample-get sentinel from the start (frozen-collection branch). A1 extends that coverage to the 3 missed callsites; no separate fix is needed for the invisible class on those paths.

### Fix shipped 2026-05-16

Five commits land on `main` the same day:

- [`bf3335d`](https://github.com/tailormemory/tailor/commit/bf3335d) — test cleanup: `FakeCollection.get()` added, 9 stale tests removed; restores a green baseline before any code change.
- [`633e089`](https://github.com/tailormemory/tailor/commit/633e089) — `verified_upsert()` grows an opt-in `retry` kwarg; `retry=0` (default) preserves the 4 batch callsites bit-identical.
- [`599acc2`](https://github.com/tailormemory/tailor/commit/599acc2) — `FakeCollection` extracted to `tests/helpers/chromadb_fake.py` so a second test file can reuse it without inter-test-file imports.
- [`7954e49`](https://github.com/tailormemory/tailor/commit/7954e49) — `kb_add`, `kb_update_session`, `/api/ingest-live` wrapped in `verified_upsert(..., retry=2)`; on failure each surfaces an explicit error to its caller (the LLM or the browser extension).
- [`f4f9e8c`](https://github.com/tailormemory/tailor/commit/f4f9e8c) — `repair_hnsw_index._classify_ghost` extended with a dispatch table for the 3 mcp_server namespaces, so the 3 known ghosts no longer block `--apply`.

### Residual state

All 5 commits are on `main`. They are **committed, not yet deployed**: the running MCP daemon imported `verified_upsert` once at boot and will not see the new code path until `launchctl kickstart` cycles it. The cycle is an operator decision, out of scope for this errata.

Recovery of the 2 `live_*` ghosts is possible by re-capturing the originating conversations via the browser extension. The `kb_add` ghost from 2026-05-12 is recoverable only if the originating LLM session is still accessible.

Canonical drift detection remains `repair_hnsw_index.py` nightly, unchanged.

### Additional generalizable lessons

**A wrapper-based fix must be verified by callsite enumeration, not by sample.** When a fix is introduced as a wrapper function, it only applies to callers that adopt the wrapper. The 4 batch ingest callsites adopted `verified_upsert()` because they were the example call sites in the original commit; the 3 mcp_server-side callsites did not, and nothing caught it for three weeks. The verification the original commit lacked was a grep for every direct `collection.add` / `collection.upsert` and a justification of why each one is or is not wrapped. A code-walk by namespace, not by example.

**A fix whose verification suite is red is a fix that has not been verified.** `tests/test_ingest_helpers.py` went 10/12 red the day v1.2.6.3 introduced the sample-get sentinel without updating the `FakeCollection` mock. Nobody noticed for three weeks because nobody ran the suite. "All tests pass" implies "someone ran them recently"; "the suite is green" is only useful information if green is the state actually observed. The standard answer is CI; in its absence, a pre-commit hook that at least runs the suites touching the changed file is the minimum.
