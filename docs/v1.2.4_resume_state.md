## v1.2.3 follow-up: verifier false-positive on transient queue lag

### What the operator saw

- Telegram end-of-run message: `🧷 HNSW drift detected: 657 chunks (email:657)`
  (matches `sync_and_ingest.log:32048`).
- `repair_hnsw_index.py --audit` run after the pipeline shows **0 SQL-only
  orphans**, 50 unrelated HNSW-only ghosts (superseded doc namespace),
  and 918 pending ops in `embeddings_queue` (oldest `2026-04-28 02:22:18`,
  newest `2026-04-28 06:09:40`).
- Pending JSON exists at
  [logs/hnsw_audits/pending_email_20260428_040014_68360.json](../logs/hnsw_audits/pending_email_20260428_040014_68360.json).

### Correction to operator hypothesis

The operator described the pending JSON as "226 entries, drift_count=None
and drift_ids=[]". **The pending JSON entries are actually populated**:
each entry has `{ids: [...], source, timestamp}` and the 226 entries
collectively hold 657 ids — exactly matching the Telegram count.

The keys `drift_count` / `drift_ids` don't exist in the pending JSON
schema; they are fields of the `maintenance_log.operation` JSON payload
written by [scripts/lib/ingest_helpers.py:124-136](../scripts/lib/ingest_helpers.py#L124-L136).
That table has the same 226 verify rows for this run, with `drift_total`
populated and `drift_ids` populated (capped at 50 per row by
`MAINTENANCE_LOG_DRIFT_ID_CAP` at
[scripts/lib/ingest_helpers.py:31](../scripts/lib/ingest_helpers.py#L31)).

So both observability paths captured the chunk ids correctly. The
operator's worry about lost ids is a misread; the real anomaly is below.

### Root cause

`verified_upsert` at [scripts/lib/ingest_helpers.py:158](../scripts/lib/ingest_helpers.py#L158)
performs:

1. `collection.upsert(...)` (writes to chromadb's metadata segment +
   enqueues the embedding into `embeddings_queue`).
2. `_vector_path_drift(...)` — single batched `collection.query(...)`
   asking each just-written chunk to be its own top-1 self-match
   ([ingest_helpers.py:43-74](../scripts/lib/ingest_helpers.py#L43-L74)).
3. If any id is not its own top-1: `time.sleep(0.5)` and retry once.
4. If still drifted: write to pending JSON + maintenance_log row.

The verifier has **no awareness of `embeddings_queue`**. ChromaDB 1.x's
local-segment architecture means upserts land in `embeddings_queue`
first; the vector segment's consumer reads from that queue and updates
the in-memory HNSW. Under bursty load, the consumer falls behind. While
chunks sit in the queue, the vector-path query won't return them — they
are functionally "missing from HNSW" until consumed.

The current state confirms chronic backpressure: 918 ops queued, oldest
from **02:22** (almost two hours of accumulation against an ingest run
that started at 03:00). Email ingest processes thousands of chunks in
quick succession; the consumer can't keep up; the verifier's 500ms
single retry doesn't outwait the lag; pending entries accumulate.

The post-pipeline audit showing 0 SQL-only orphans confirms the chunks
*are* eventually consumed — the drift is transient, not permanent.

Operator's hypothesis (transient queue lag misread as permanent drift)
is correct. The 500ms / single-retry value at
[ingest_helpers.py:30](../scripts/lib/ingest_helpers.py#L30) was tuned
against a non-email workload and underestimates the email burst pattern.

### Severity

| Dimension                  | Impact |
|----------------------------|--------|
| Data correctness           | **Safe.** Pending entries are non-destructive — they queue chunks for the next audit. The `repair_hnsw_index.py` audit re-checks live state and confirms zero orphans, so no spurious re-embedding happens. |
| Repair-side cost           | **None.** Pending entries do not auto-trigger `--apply`; that requires manual operator action with maintenance mode + recent backup. |
| Alerting fidelity          | **Degraded.** Telegram fires "HNSW drift detected" every email-heavy run when there is no actual drift. Repeated false alarms train the operator to ignore the signal — when real drift eventually happens, it will be missed. |
| Storage                    | Negligible. The pending JSON is ~150 KB; maintenance_log gains 226 rows/run. |

### Proposed fixes (NOT IMPLEMENTED — operator decides)

1. **Queue-membership check before recording drift** (preferred — small,
   structural, eliminates the false positive entirely). After
   `_vector_path_drift` returns a non-empty list and the 500ms retry
   still fails, query `embeddings_queue` for those ids and drop any
   present. Only ids genuinely missing from both HNSW and the queue are
   permanent drift. ~15 lines in `verified_upsert`, one extra SQL
   `SELECT ... WHERE id IN (...)`.
2. **Lengthen retry to a small loop** (cheap fallback if #1 is rejected).
   Replace single 500ms retry with e.g. 3 retries at 500ms / 1s / 2s.
   Accepts ~3.5s worst-case latency per drifted batch but doesn't fix
   the deeper "verifier doesn't know about the queue" issue.
3. **Skip verification when queue is large** (escape hatch). At top of
   `verified_upsert`, peek `SELECT COUNT(*) FROM embeddings_queue`. If
   above N (suggest 200), skip the read-back and log a single warning
   per run instead of recording per-batch drift. Cheap, but loses
   verification entirely during bursts.
4. **Documentation/UX cleanup.** The pending JSON entry key is `ids`
   while the maintenance_log row key is `drift_ids`. Renaming pending
   JSON to `drift_ids` would have prevented the operator's misread. Trivial,
   but a schema change — flag it for migration.

### Recommendation: hotfix v1.2.3.1 vs roll into v1.2.4

**Recommend hotfix v1.2.3.1** with fix #1 (queue-membership check).
Reasoning:

- Data path is safe, but the alerting channel is being eroded with each
  pipeline run. Every additional night with false alerts further
  desensitises the operator to a real drift event.
- The fix is small and self-contained inside `ingest_helpers.py` — no
  cross-module surface area, no schema change.
- Bundling into v1.2.4 risks the v1.2.4 scope dragging while operator
  loses confidence in the verifier signal.

If a hotfix is undesirable for release-cadence reasons, the second-best
option is to put fix #3 (queue-size escape hatch) into v1.2.4 first as a
mitigation, with fix #1 following. Do **not** roll a fix that only
extends retry duration (#2 alone) — it papers over the symptom and the
500ms→multi-retry value will need re-tuning every time corpus size
changes consumer throughput.

## Pipeline cost diagnostic — why Anthropic spend hasn't dropped

### What the operator saw

- Anthropic spend last night: ~$20.
- Forecast at steady state: $0.50–1.50/day.
- KB stats: 155,632 chunks total; 155,626 with `entities_extracted=1`
  ("100% coverage"); supposed backlog "exhausted".
- Telegram report: `+23046 fatti (2486 chunk)` despite the 100% figure.
- Pipeline duration: 16,204 s (4 h 30 min) vs expected steady-state
  duration of minutes.
- Supersession reported as "TIMEOUT, zero work completed but tokens
  consumed".

### Root cause #1 — the "100% coverage" figure measures the wrong thing

The operator's "155,626 chunks with `entities_extracted = 100% coverage"
is metadata written by `extract_entities.py` (entity-tag extraction).
**This is unrelated to fact extraction.** Fact extraction tracks coverage
in `facts.sqlite3.extraction_log`, not in chroma metadata.

Verified counts (queried directly):

- `chroma.sqlite3` chunks with `entities_extracted=1` (int_value): **155,626** ✓
  (operator's number is correct, but it's the entity-extraction marker)
- `facts.sqlite3.extraction_log` rows: **81,470**
- Fact-extraction coverage: **81,470 / 155,633 ≈ 52%**

So roughly **74,000 chunks still have no fact extraction**. The 2,486
chunks processed last night were drawn from this backlog — that's the
expected "chunks not in extraction_log" path at
[extract_facts_nightly.py:269-321](../scripts/enrichment/extract_facts_nightly.py#L269-L321).

The pipeline is not at steady state for fact extraction. At ~2,500
chunks/night absorbed (limited by the timeout, not the backlog), the
remaining backlog will take ~30 nights to drain.

### Root cause #2 — Anthropic Haiku is the FIRST backend in rotation, not the fallback

[config/tailor.yaml](../config/tailor.yaml) `enrichment.fact_extraction.backends`
lists:

```
- provider: anthropic    (claude-haiku-4-5-20251001)  workers: 15
- provider: google       (gemini-2.5-flash)           workers: 5
- provider: openai       (gpt-4o-mini)                workers: 8
daily_limit_per_backend: 4500
```

`BackendManager` at [extract_facts_nightly.py:113-169](../scripts/enrichment/extract_facts_nightly.py#L113-L169)
rotates strictly in list order. With 4,500 calls/backend/day and a
2,486-chunk batch, the rotation **never reaches Gemini or OpenAI** —
Anthropic absorbs every fact-extraction call. The `extraction_log` model
distribution since 2026-04-27 confirms this: 4,787 rows with
`model='anthropic'`, 0 with `model='google'`, 0 with `model='openai'`.

`fact_supersession` and `fact_derivation` are configured Anthropic-only
in the same YAML (no fallback list). Combined with the
backlog-not-exhausted finding above, this is the cost driver.

### Root cause #3 — supersession is timing out because the corpus has scaled past the budget

Pipeline timeline (from [logs/sync_and_ingest.log](../logs/sync_and_ingest.log)
lines 31292–32048):

| Step | Wall time | Result |
|------|-----------|--------|
| Sync + ingest + entities + summaries + profile | 03:00:01 → 03:02:46 (~3 min) | normal |
| Fact extraction | 03:02:46 → 05:30:19 (147 min) | TIMEOUT after 145 min, 2,486 chunks → 23,046 facts |
| Fact supersession | 05:30:19 → 07:00:21 (90 min) | TIMEOUT after 88 min, **18,400 of 50,005 pairs processed**, SUP=95 |
| Fact derivation | 07:00:21 → 07:30:03 (30 min) | normal, 1,623 facts derived from 535 entities |

**Operator's "supersession went TIMEOUT (zero work completed)" claim is
incorrect.** The Telegram message reported 0 because
[sync_and_ingest.sh:404-407](../sync_and_ingest.sh#L404-L407) extracts
"Coppie confrontate:" via grep — that line only prints in the **final
summary block** at [fact_supersession.py:443](../scripts/enrichment/fact_supersession.py#L443),
which never executes when the script is killed by SIGTERM during the
processing loop. The actual log shows progress to `[18,400/50,005]` at
3.4 pairs/s with SUP=95 right before the kill (line 31495). Each batch
commits every 20 pairs ([fact_supersession.py:428-429](../scripts/enrichment/fact_supersession.py#L428-L429)),
so the 95 supersessions are durable.

(Side note: a DB query for `superseded_at >= '2026-04-27'` returns 37,
not 95. The 58-fact gap is consistent with mid-batch SIGTERM losing
in-flight comparisons before commit. This is a separate small bug worth
noting.)

The supersession runtime explosion is structural, not a regression. The
algorithm at [fact_supersession.py:98-190](../scripts/enrichment/fact_supersession.py#L98-L190)
loads ALL non-superseded facts (currently 921,085 in
`facts.sqlite3`), groups by (category, primary_entity), and generates
candidate pairs within each group. Pair count grows roughly with O(n²)
per group; at 920k facts it produces 50,005 candidates against the
`MAX_COMPARISONS` cap. At 3.4 pairs/s on Haiku, exhausting the cap
takes ~4 h — far over the 90-min budget at
[sync_and_ingest.sh:131](../sync_and_ingest.sh#L131).

### Cost breakdown estimate (Haiku 4.5 at ~$1/M input, ~$5/M output)

| Step | Calls | Approx. tokens | Approx. cost |
|------|-------|----------------|--------------|
| Fact extraction | 2,486 chunks × 1 call | ~3.2k in / ~450 out per call | **~$13–14** |
| Fact supersession | 18,400 pair comparisons | ~250 in / ~30 out per call | **~$5–7** |
| Fact derivation | 535 entities, ~3 facts/entity | small prompts | **~$1–3** |
| User profile | 1 call | one large prompt | **~$0.10** |
| Doc summaries | 0 (none generated last night) | — | $0 |
| **Total estimate** | | | **~$19–24** |

This brackets the observed $20. Fact extraction alone is the largest
single line item.

The "tokens consumed during timeout" hypothesis from the prompt is
**partially right but mis-attributed**: each Haiku call is a complete
non-streaming request, so calls that finish before the SIGTERM are
fully billed and produce useful work. Calls in flight at SIGTERM are
also billed but their results are dropped (this is the 95 vs 37 gap
above). It is not the case that supersession billed $5 for "zero work";
it billed for ~18,400 useful comparisons that mostly committed.

### Severity

| Dimension                | Impact |
|--------------------------|--------|
| Cost overrun             | $18–22/day above forecast (~$540/month). Material. |
| Operator trust           | Forecast was based on wrong coverage metric — will recur unless dashboard fixed. |
| Pipeline reliability     | Two of three enrichment steps timing out nightly; supersession timing out cleanly, extraction silently capping at ~2,500/night against a 74k backlog. |
| Data correctness         | Safe — timed-out calls produce no bad data, just truncated nightly progress. The 95→37 commit-gap is small, recurs at low rate, recovers next night. |

### Proposed fixes (NOT IMPLEMENTED — operator decides)

1. **(URGENT) Move `fact_supersession.py` to Anthropic Batch API.**
   The pair comparisons are independent (no inter-pair state). The Batch
   API is ~50% cheaper than the synchronous endpoint, has 24h SLA which
   fits a nightly pipeline, and removes the 90-min wall-clock pressure
   entirely. This was already in the v1.2.4 backlog per
   [docs/supersession-requirements.md](supersession-requirements.md);
   the cost numbers above make it the highest-leverage v1.2.4 item.
2. **(URGENT) Add a fallback chain to
   `enrichment.fact_supersession.backends` and `fact_derivation.backends`
   in [config/tailor.yaml](../config/tailor.yaml).** The COMPARISON_PROMPT
   is binary classification — Gemini Flash handles it cleanly at
   ~10× lower cost. Same for derivation. Configuration-only change, no
   code edit. Could be a same-day mitigation pre-v1.2.4.
3. **Stop using `entities_extracted` as the coverage proxy in any
   operator-facing dashboard.** The Telegram message and any wiki/dash
   should report `extraction_log` row count vs total chunks. The
   operator's $0.50–1.50/day forecast was anchored to the wrong metric;
   without this fix the forecast keeps drifting from reality.
4. **Track `supersession_checked_at` per fact** so supersession only
   re-runs on facts whose group has changed since last check. Also
   already in scope per supersession-requirements.md; combined with #1
   should bring nightly supersession well inside budget.
5. **Add a SIGTERM handler to
   `fact_supersession.py`** that flushes the in-flight batch before
   exit. Closes the 95→37 commit gap. ~10 lines, but secondary
   priority — the work isn't lost-lost (those pairs will recur as
   candidates next night), just deferred.
6. **Reconsider the backend ordering for `fact_extraction`.** Putting
   Anthropic first means cost is paid before cheaper fallbacks are
   reached. If Anthropic is preferred for *quality* on extraction
   (judgement call worth checking), keep it; otherwise reorder
   Gemini → OpenAI → Anthropic to match the rotation pattern documented
   in the script's docstring.

#1 + #4 the v1.2.4 anchor.
