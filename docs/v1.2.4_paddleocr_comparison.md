# PaddleOCR Backend — v1.2.4 Validation Report

## Executive summary

TAILOR v1.2.4 introduces an optional second extraction backend for PDF
documents based on PaddleOCR's `PPStructureV3` pipeline (table-aware OCR with
HTML/markdown output). The pre-existing pipeline — native PyMuPDF text
extraction → Tesseract OCR fallback → Apple Vision OCR fallback — remains the
default for all routed-out doctypes and the safety net for every routed-in
document.

This report documents the four-stage probe campaign (Steps 4a / 4a.5 / 4a.6 /
4a.7) used to evaluate PaddleOCR on real Italian financial and tax paperwork
in this repository's domain. Per-doctype verdicts diverge sharply: bank
statements show high structural binding quality (64.7% of monetary rows
preserved cleanly) and benefit from paddle; "centro di costo" management
balance sheets degrade catastrophically (column fusion destroys row-level
bindings) and are now blocked from paddle by content-based detection;
Italian Revenue Agency (AdE) tax modules emit `<tr>` rows but cannot
preserve label-value bindings inside them, so paddle is useful only as
"better OCR" producing a flat text re-OCRing rather than as a structured
extractor; XBRL-style narrative notes show no measurable advantage either
way.

The architectural decision is: **paddle is enabled in config but routed
conservatively**. A document reaches paddle only when (a) its `doc_type`
matches the configured whitelist, or (b) the native text layer is below
threshold (suggesting a scan), and never when content-based detection flags
it as a known-broken template family. Whole-doc and per-page failures fall
back to the legacy pipeline with a JSON audit trail. Models are cached
locally after a one-time ~1 GB download from Paddle's CDN; per-page predict
latency on Apple Silicon CPU sits at roughly 19 seconds for typical pages
and grows with text density.

### `PPStructureV3` initialization

All probes used the same kwargs profile (validated in Step 4a):

```python
PPStructureV3(
    lang="it",
    use_textline_orientation=True,
    use_table_recognition=True,
    use_seal_recognition=False,
    use_formula_recognition=False,
    use_chart_recognition=False,
    use_doc_orientation_classify=False,
    use_doc_unwarping=False,
)
```

Step 4a.6 validated that `use_doc_orientation_classify=True` and
`use_doc_unwarping=True` do not improve the gestionale failure mode and
introduce regressions on prose-heavy markdown; they are intentionally left
off in the production initializer. See "Architectural decisions" below.

The recognizer model selected by `lang="it"` is
`latin_PP-OCRv5_mobile_rec`. The corresponding server-grade recognizer
(`latin_PP-OCRv5_server_rec`) was not adopted in v1.2.4 — see "Open
questions".

### Predict signature

```python
result = pipeline.predict(input=filepath)
pages = list(result)  # materializes a list of LayoutParsingResultV2
```

Each `LayoutParsingResultV2` exposes:

- A `Mapping` interface with keys including `parsing_res_list`,
  `overall_ocr_res`, `model_settings`, `page_index`, `page_count`.
- Attribute accessors for output formats: `markdown` (returns a `dict`
  with key `markdown_texts`), `json`, `html`, `latex`.

The TAILOR adapter (`scripts/ingest/ingest_docs.py::_extract_pdf_paddle`)
prefers `overall_ocr_res["rec_texts"]` for the flat-text payload and falls
back to a HTML-stripped projection of `markdown["markdown_texts"]` if that
is empty.

### Evaluation criterion: binding ratio

For tabular documents, "did paddle preserve table structure?" was reduced
to a single quantitative metric, **binding ratio**:

```
binding_ratio = clean_rows / rows_with_value

where:
  rows_with_value  = number of <tr> rows containing at least one monetary
                     value matching r"\b\d{1,3}(?:\.\d{3})*,\d{2}\b"
                     (Italian formatting)
  clean_rows       = subset of those rows containing exactly one monetary
                     value AND at least one non-numeric label token
```

A clean row is one where a downstream consumer can confidently associate
the single value to the row's label. Rows with multiple values clumped in
the same `<td>` (the failure mode observed on gestionali) are excluded.

Verdict labels:

- **intact**: ≥ 70% of monetary rows have a clean 1-value+label binding
- **degraded**: 30–70% — partial structure preserved, post-processing required
- **broken**: < 30% — table structure unreliable for binding
- **no_tables**: paddle did not emit `<tr>` rows for this document

These are heuristic thresholds, intended for routing decisions rather than
formal model evaluation.

## Detailed findings per doctype

### Bilancio gestionale centro-di-costo (e.g. YOUniversal Media)

**Layout description.** A2-landscape A4 sheet rendered by Italian
accounting software (Sistemi/AGO product family — the page footer of the
sample reads `AGO - 05.15.01 Utente: user01 Data: 11-11-2024 Ora: 16:33`).
The page is a side-by-side two-column ledger: left column lists COSTI
account codes with amounts, right column lists RICAVI account codes with
amounts. There are no visible vertical rules between cells — column
boundaries are spatial, not graphical, which appears to be the underlying
reason for layout detection failures (see below).

**Failure mode.** PaddleOCR's table recognizer correctly identifies that a
table is present, but mis-segments the cells: for the RICAVI block, it
fuses five distinct monetary values from five distinct account-code rows
into a single `<td>` element, then emits the corresponding five account
codes as a separate `<td>` in another row. Row identity is lost.

**Paddle markdown output (page 0, RICAVI column, abbreviated):**

```html
<tr>
  <td>Centro di costo 084 - Agosto 2024</td>
  <td></td><td></td><td></td>
  <td>Regime Contabile 219,90 98.445,04 2.962,52 62.503,89 25,00</td>
  <td>Ordinario 500260 000 - prestazioni di servizi Italia
      500265 002 - prestazioni di servizi marketing BD
      500265 003 - prestazioni di servizi assistenza IT
      500270 000 - prestazioni di servizi extra UE</td>
  <td>Tipo Reddito Impresa
      500265 001 - prestazioni di servizi intra UE Intercompany</td>
  <td></td>
</tr>
```

All five values (`219,90 / 98.445,04 / 2.962,52 / 62.503,89 / 25,00`) are
in one `<td>`. Account codes are split across two adjacent `<td>` cells in
no particular order. There is no way to recover the binding
`account_code → amount` from this row.

**Legacy PyMuPDF flat text (same page, same region):**

```
500260 000 - prestazioni di servizi Italia
                  219,90
500265 001 - prestazioni di servizi intra UE Intercompany
98.445,04
500265 002 - prestazioni di servizi marketing BD
2.962,52
500265 003 - prestazioni di servizi assistenza IT
62.503,89
500270 000 - prestazioni di servizi extra UE
25,00
```

Each account code appears immediately before its amount in the natural
reading order. A simple line-by-line consumer can recover the bindings
without any layout reasoning.

**Step 4a.6 outcome.** Toggling `use_doc_orientation_classify=True` did
not change the column-fusion behavior (RICAVI values still clumped); page
counts and binding ratios were unchanged. Toggling
`use_doc_unwarping=True` further shortened the markdown payload from 4617
to 3695 chars on page 0 and 2145 → 1911 on page 1, suggesting it
suppressed content rather than recovering structure. Both options were
left at `False`.

**Verdict.** Routed to legacy via the `_is_bilancio_gestionale()` content
detector (see "Architectural decisions"). Paddle is not invoked at all
for this template family.

### Estratto conto bancario

**Layout description.** Multi-page bank statement with a per-row schema:
`data operazione | data valuta | importo a debito | importo a credito |
descrizione operazione`. Vertical column boundaries are graphical, with
some descriptive text continuing into subsequent rows in narrower
formatting.

**Failure mode.** None at the structural level. The table recognizer
correctly preserves one `<tr>` per movement, with the date, value-date,
amount, and description columns intact. Some descriptive lines that span
two rows in the source are kept in two `<tr>` rows with empty leading
columns, which is faithful to the visual layout.

**Paddle markdown output (page 1, three movement rows):**

```html
<tr>
  <td>27/10/2022</td><td>31/10/2022</td>
  <td>3.000,00</td><td></td>
  <td>PAGAMENTO PERIODICO mantenimento giovanna anello</td>
</tr>
<tr>
  <td>04/11/2022</td><td>04/11/2022</td>
  <td>713,00</td><td></td>
  <td>BONIFICO A DEBITO</td>
</tr>
<tr>
  <td>07/11/2022</td><td>07/11/2022</td>
  <td></td><td>4.006,00</td>
  <td>STIPENDIO</td>
</tr>
```

Date, debit/credit column, and operation type are all bound to the same
row. Downstream consumers can build `(date, amount, sign, description)`
tuples directly.

**Legacy PyMuPDF flat text (same page, equivalent region):**

```
27/10/2022  31/10/2022         3.000,00                       PAGAMENTO PERIODICO
                                                              mantenimento giovanna anello
                                                              anello giovanna
04/11/2022  04/11/2022           713,00                       BONIFICO A DEBITO
                                                              a favore di condominio via francesco crispi
07/11/2022  07/11/2022                            4.006,00    STIPENDIO
                                                              youniversal media srl
                                                              retribuzione mese ottobre 2022
```

The legacy output relies entirely on whitespace alignment to convey
column position. A consumer must implement column-detection heuristics
(usually fragile across statement formats) to reconstruct what paddle
delivers as structured cells.

**Verdict.** PaddleOCR is the better backend for this doctype. 64.7% of
movement rows are in clean `(label, single_value)` form straight out of
predict. By the strict threshold this is "degraded", but it is the
strongest binding-ratio signal observed in this validation.

**Routing note.** As of v1.2.4, an estratto conto bancario is **not**
routed to paddle by default. The `infer_doc_type()` classifier in
`scripts/ingest/ingest_docs.py` does not currently emit a
`"estratto_conto"` label that matches the `paddleocr.structured_doc_types`
whitelist, and bank statements have a healthy native text layer (so the
`min_text_threshold` low-text fallback also does not fire). Realizing this
gain in production requires extending the doctype classifier — see "Open
questions".

### Modulistica AdE (Unico PF, 730, dichiarazioni)

**Layout description.** Italian Revenue Agency forms are pre-printed
templates with named field codes (`RP1`, `RP7`, `RN1`, etc.) arranged on
multi-column form sheets. The visual structure is dense: many field labels
have placeholder thousand-separator markers (`,00`) for unfilled values,
field codes are typeset adjacent to but not always inside the same
graphical cell as their values, and individual cells often span multiple
columns or rows.

**Failure mode.** Paddle correctly identifies that pages contain tables
and emits a large number of `<tr>` rows (396 across the 20-page sample).
However, only **2** of those rows contain a monetary value at all, and
only **1** has a clean label-plus-single-value binding. The remaining
rows are either empty visual scaffolding (`<td>2 ,00</td>` for unfilled
fields), heavily nested with `colspan/rowspan` chaining, or mix
descriptive prose with numeric placeholders.

**Paddle markdown output (page 7, Quadro RP "Oneri e spese", abbreviated):**

```html
<tr>
  <td>QUADRO RP ONERI E SPESE Sezione I Spese per le quali spetta la
      detrazione d'imposta del 19%, del 26% del 30%, del 35% e del 90%
      Le spese mediche vanno indicate interamente senza sottrarre la
      franchigia di euro 129,11 ...</td>
  <td colspan="4">RP1 Spese sanitarie</td>
  <td>Spese patologie eșenti sostenute da familigri</td>
  <td>Spese sanitarie comprensive di franchigia euro 129,11 2 3.605,00</td>
  <td>,00</td>
</tr>
<tr>
  <td>RPInteressi mutui ipotecari acquisto abitazione principale</td>
  <td colspan="4">4.000,00</td>
  <td>2 ,00</td>
  <td>RP8 Altre spese Codice spesa</td>
  <td>36</td>
</tr>
```

The field code `RP7` was OCR'd as `RPInteressi` (the digit 7 was lost into
the word "Interessi" — the recognizer's confidence on the small digit was
low). The value `4.000,00` is in the row, but the field-code-to-value
binding requires post-processing to recover.

**Useful signal in flat text.** Although the table structure is
unreliable, the OCR'd flat text for the same page does contain the field
codes and their values, in spatial order. Stripping HTML and counting
Italian-formatted monetary literals (`\b\d{1,3}(?:\.\d{3})*,\d{2}\b`)
across all 20 pages yields **57 occurrences** of plausible field amounts
(38 unique). Examples within ~80 characters of an `RP\d+` code on page 7:

| Field code | Value | Source line context |
|------------|------:|---------------------|
| RP14 | 7.476,00 | Spese per canone di leasing |
| RP22 | 24.000,00 | Assegno al coniuge (CF coniuge present) |
| RP23 | 461,00 | Contributi servizi domestici |
| RP27 | 5.160,00 | Previdenza complementare |
| RP39 | 29.621,00 | TOTALE ONERI E SPESE DEDUCIBILI |
| RP48 | 8.890,00 | Rata 50% interventi recupero |
| RP49 | 4.445,00 | TOTALE DETRAZIONE |

Page 9 exposes `RN1 Reddito complessivo = 91.593,00`. So the values are
recoverable in principle, but only if a regex- or LLM-based label-value
post-processor runs over the flat text. Paddle's `<tr>` output is not the
right unit of recovery here.

**Legacy PyMuPDF flat text (same page).** PyMuPDF's text layer extraction
on the AdE PDF is also non-trivial: AdE forms are typically published as
filled-form PDFs where field labels and field values are placed in
separate stream objects, so reading order is template-defined rather than
visual. Sample of native text from page 7:

```
QUADRO RP
ONERI E SPESE

Sezione I
Spese per le quali
spetta la detrazione
d'imposta del 19%, del 26%, ...

RP1
RP3
RP2
RP4
...
```

Field codes and values appear in two separate streams; binding by visual
proximity is harder than with paddle's flat OCR text where label and
value are close together.

**Verdict.** Paddle is **useful as enhanced OCR** for AdE forms (it
produces cleaner flat text than PyMuPDF's stream-based reading order)
**but is not a structured extractor** for this doctype. v1.2.4 does not
ship a binding post-processor; AdE module support is therefore best-effort
ingestion only. Field-level structured extraction is left as an open
question.

**Routing.** AdE modules are not in the default `structured_doc_types`
whitelist either; if added later, they should arrive together with a
post-processor.

### Nota integrativa XBRL

**Layout description.** Mostly narrative prose (multi-page accounting
notes), with a small number of summary tables embedded. The XBRL
inline-rendered version of the document is what was tested.

**Verdict.** No measurable advantage either way. Paddle and legacy
produce equivalent flat-text content for the prose sections; on the few
embedded tables, paddle's `<tr>` output binds cleanly (1/1 = 100% in the
sample) but the tabular volume is too small to draw conclusions.

**Paddle markdown output (page 4, prose excerpt):**

```
energetiche, elemento oggi più che mai centrale, a causa dei forti shock
causati prima di tutto dalla nascita del conflitto in Ucraina, con le
relative sanzioni alla Russia, e poi dallo scoppio del tumulto in Medio
Oriente, che ha avuto anch'esso un forte impatto sul prezzo di tali
materie prime. ...
```

**Legacy PyMuPDF flat text (same page, prose excerpt):**

```
energetiche, elemento oggi più che mai centrale, a causa dei forti shock
causati prima di tutto dalla nascita del conflitto in Ucraina, con le
```

## Architectural decisions

### Init kwargs (PPStructureV3)

The production initializer in
`scripts/ingest/ingest_docs.py::_get_paddle_ocr` uses:

| Kwarg | Value | Rationale |
|-------|-------|-----------|
| `lang` | `"it"` | Selects `latin_PP-OCRv5_mobile_rec`; matches the document language. |
| `use_textline_orientation` | `True` | Required for rotated lines that occasionally appear in scans. |
| `use_table_recognition` | `True` | The whole point of the experiment is structured table extraction. |
| `use_seal_recognition` | `False` | No corporate seal detection needed; saves a model. |
| `use_formula_recognition` | `False` | No LaTeX/formula detection needed. |
| `use_chart_recognition` | `False` | No chart-to-data extraction needed. |
| `use_doc_orientation_classify` | `False` | Tested in Step 4a.6 against the rotated bilancio gestionale; did not change the column-fusion failure. Adds a model load and a small per-page overhead with no observed benefit on the production sample set. |
| `use_doc_unwarping` | `False` | Tested in Step 4a.6; markdown payload shrank by ~15–20% with content suppression rather than structural improvement. Adds the `UVDoc` model download and a measurable init slowdown (Step 4a.6 measured init 1.68 s → 15.00 s). |

The recognizer model selected by `lang="it"` is the **mobile** variant
(`latin_PP-OCRv5_mobile_rec`). The server variant is more accurate per
PaddleOCR's published evaluation but 2–3× slower at predict time on CPU.
Given the per-page latency floor on Apple Silicon CPU (see "Performance"),
v1.2.4 keeps the mobile recognizer. The server variant is not configured.

### Performance

Numbers are from `/tmp/probe_paddle_warm_output.txt` (Step 4a.5) on
Apple Silicon Mac Mini M4, CPU. Use of MPS (Metal Performance
Shaders) acceleration on Apple Silicon was not investigated in v1.2.4 —
these timings reflect the default CPU execution path. Operators with
throughput concerns may explore Paddle's MPS support, but its
compatibility with `PPStructureV3` at the pinned version is unverified by
this report.

| Operation | Time | Notes |
|-----------|-----:|-------|
| `import paddleocr.PPStructureV3` | 1.74 s | Dominated by PaddlePaddle native lib load |
| `PPStructureV3(**kwargs)` warm init | 1.60 s | All 11 model packs already present in `~/.paddlex/official_models/` |
| `pipeline.predict(input=...)` first call (2 pages) | 37.87 s | 18.93 s/page |
| Same, second call on a different file (2 pages) | 38.28 s | 19.14 s/page |
| Same, third call repeating the first file (2 pages) | 38.22 s | 19.11 s/page |

For higher-density documents (Step 4a.7), per-page latency rises:

- Estratto conto: 25.04 s/page (9 pages, 225.36 s total)
- Unico PF: 32.39 s/page (20 pages, 647.90 s total)
- Nota integrativa: 20.54 s/page (20 pages, 410.70 s total)

The variation tracks with text density — heavier pages need more OCR
inference per `<tr>`/cell. The ~19 s/page figure should be read as a
floor for typical document pages on M4 CPU, not a guaranteed throughput.

**Cold-init cost.** On the first PPStructureV3 instantiation against an
empty `~/.paddlex/` cache, the pipeline downloads 11 model packs from
Paddle's CDN totaling **957.8 MB**. The Step 4a probe measured
**~52 minutes** wall-clock for the full download on a residential
connection in Italy. Individual downloads ranged from a few seconds to
17+ minutes for the largest pack. The cache survives reboots; subsequent
runs warm-init in 1.6 s.

**Implication for nightly batch ingestion.** Routing N documents to
paddle within a single process amortizes the 1.6 s init cost across all
N and pays only the ~19–32 s/page predict cost. The module-level
singleton (`_paddle_pipeline` in `_get_paddle_ocr()`) ensures this
reuse. In a process-per-doc scheduler, every invocation pays the warm
init cost; that is acceptable but not optimal. v1.2.4 does not include a
long-running paddle worker process; this is left to the operator's batch
strategy.

### Routing logic

`scripts/ingest/ingest_docs.py::extract_pdf` selects between paddle and
legacy as follows:

1. If `cfg("paddleocr", "enabled", False)` is `False`, route everything
   to legacy. (Default in `config/tailor.yaml.example` until operators
   opt in.)
2. Compute `doc_type = infer_doc_type(filename)`.
3. Call `_route_to_paddle(filepath, doc_type)` which returns
   `(should_route: bool, reason: str | None)`.
4. The routing decision is:
   - **gestionale guard (negative whitelist)**: if
     `_is_bilancio_gestionale(filepath)` is `True` → return
     `(False, "gestionale_blocklist")`. PaddleOCR never sees the file.
   - **doctype whitelist**: if
     `doc_type in cfg("paddleocr", "structured_doc_types", [])` →
     return `(True, "doc_type")`.
   - **low-text fallback**: if the sum of native text lengths across
     all pages is below
     `cfg("paddleocr", "min_text_threshold", 100)` → return
     `(True, "low_text")`. This handles scanned PDFs where the native
     pipeline would produce nothing useful.
   - otherwise → `(False, None)`.

The gestionale guard runs first because it is a known-broken case for
paddle; it should override any positive routing decision the operator
might configure. The check is implemented via PyMuPDF first-page text
extraction with a small set of pattern matches:

```python
_GESTIONALE_PATTERNS = (
    "BILANCIO DI VERIFICA",
    "TCA PER CENTRO DI COSTO",
    "Centro di costo",
    "TCA per centro di costo",
)
```

These match the Sistemi/AGO ledger family observed in the validation
sample. The detector is conservative: any extraction failure (corrupt
PDF, password-protected, etc.) returns `False` so paddle still runs and
gets a chance to fall back via its own error path.

### Fallback hierarchy

PaddleOCR-routed documents have three layers of fallback:

| Failure point | Action | Audit |
|---------------|--------|-------|
| `_get_paddle_ocr()` raises (init failure: missing model, OOM, etc.) | Whole-doc → `_extract_pdf_legacy(filepath)`, normalized to paddle's section schema via `_normalize_legacy_sections()` | `logs/paddle_fallback_<YYYYMMDD>.json` entry with `reason="init_failure"` |
| `pipeline.predict(...)` raises at top level | Whole-doc → `_extract_pdf_legacy(filepath)`, normalized | `reason="whole_doc_predict_failure"` |
| `predict` returns 0 pages | Whole-doc → legacy, normalized | `reason="zero_pages"` |
| Per-page processing raises (e.g., a single corrupt page) | Lookup that page in the lazily-extracted legacy result by 1-indexed page number; if found, wrap with `extractor="legacy_fallback"`; if not found, emit empty placeholder with `extractor="legacy_fallback_empty"` | Single batched `reason="per_page_fallback"` entry with one `events[]` item per failed page |
| Per-page legacy fallback also raises | Empty placeholder with `extractor="double_failure"` | Logged inline in the same per-page entry |

Audit logs are appended to `logs/paddle_fallback_<YYYYMMDD>.json` as a
JSON array. Each entry has:

```json
{
  "ts": "<ISO timestamp>",
  "filepath": "<absolute path>",
  "reason": "init_failure | whole_doc_predict_failure | zero_pages | per_page_fallback",
  "error": "<exception message or 'see events'>",
  "events": [{"page": <int>, "error_type": "<class>", "error": "<msg>"}, ...]
}
```

This gives operators a single place to look when paddle silently
"mostly works": if a document's downstream search recall regresses, the
audit log will show whether paddle handled it cleanly or fell back on
some pages.

### Section schema

`_extract_pdf_paddle` returns a list of dicts with the following shape
(consistent across paddle output, whole-doc fallback, and per-page
fallback):

```python
{
    "text": <flat text, backward-compatible with legacy consumers>,
    "markdown": <HTML/markdown with table structure, may be empty>,
    "metadata": {
        "page": <1-indexed integer>,
        "total_pages": <int>,
        "ocr": True,
        "extractor": "paddleocr"
                  | "legacy_fallback"
                  | "legacy_fallback_empty"
                  | "double_failure",
    },
}
```

The 1-indexed page numbering matches `_extract_pdf_legacy`'s existing
convention so that consumers downstream of `extract_pdf()` see uniform
metadata regardless of which backend produced a section.

The `markdown` field is **new in v1.2.4** and is empty string (`""`) on
legacy-produced sections. Existing consumers that expect only
`text` + `metadata` continue to work; consumers that want table
structure can opt in to the markdown payload.

The `extractor` metadata field allows downstream code (and the search
backend) to reason about which pipeline produced any given section,
which is useful for retrieval-quality analysis post-deployment.

## Limits & known issues

- **Bilanci centro-di-costo gestionali are routed to legacy.** This is
  the configured behavior, not a regression. Documents in this template
  family produce structurally broken paddle output (column fusion); the
  `_is_bilancio_gestionale()` content detector blocks them. Operators
  who want paddle to attempt these documents anyway can disable the
  detector by editing the production code (no config flag is provided
  in v1.2.4 — this is intentional, since the detector encodes a
  validated quality verdict).

- **Modulistica AdE: `<tr>` structure is unreliable.** Paddle emits
  table rows on AdE forms but the rows are mostly empty visual
  scaffolding. Field-level value recovery requires a flat-text
  post-processor that v1.2.4 does not ship. Until then, AdE modules can
  be ingested for full-text search but field-level retrieval is not
  guaranteed. AdE modules are not in the default routing whitelist, so
  this is currently a non-issue in practice — they are handled by
  legacy.

- **1 GB model download on first use.** PaddleOCR's `PPStructureV3` does
  not ship a single-file offline package. First initialization triggers
  ~957.8 MB of model fetches from Paddle's CDN, taking ~52 minutes on a
  typical residential connection. Operators bringing TAILOR up offline
  should pre-warm the cache by running a single
  `_get_paddle_ocr()` invocation in a network-enabled environment and
  then copying `~/.paddlex/official_models/` to the target machine.

- **Predict latency is ~19–32 s/page on M4 CPU.** This is a hard floor
  for the mobile recognizer on Apple Silicon. The server recognizer
  would be 2–3× slower; GPU/MPS acceleration on Apple Silicon was not investigated in v1.2.4. For a
  typical 20-page tax return, this means ~6–11 minutes of predict time.
  Routing should reflect this: paddle should not be invoked on
  documents that would not benefit from it.

- **Mobile recognizer accuracy.** `latin_PP-OCRv5_mobile_rec` is
  optimized for speed on Latin-script documents. The validation sample
  showed isolated OCR errors (e.g., the field code `RP7` rendered as
  `RPInteressi` on the Quadro RP page — the digit 7 collapsed into the
  adjacent word). Such errors are inherent to mobile-class OCR and
  unlikely to be eliminated without switching recognizer.

- **Audit log file rotation is by date, not size.** A high-volume
  ingestion day with many fallbacks could grow
  `logs/paddle_fallback_<YYYYMMDD>.json` arbitrarily. v1.2.4 does not
  cap or rotate this file. For typical TAILOR usage (small personal
  archives), this is not expected to matter; large deployments should
  add a log-management strategy.

## Open questions for v1.2.5+

These are not committed roadmap items; they are observations from this
validation that future releases should consider.

- **`infer_doc_type()` coverage.** As of v1.2.4, the doctype classifier
  does not emit a stable label for bank statements
  (`"estratto_conto"`-class), and the validation report identified
  estratto conto as paddle's clearest win. Realizing that win in
  production requires the classifier to fire reliably. Reviewing the
  classifier's coverage against the documents that actually populate
  `cfg("paddleocr", "structured_doc_types")` is the most direct path
  to enable paddle for the doctype where it is most useful.

- **Vision LLM backend (e.g. Gemini Flash, Claude Sonnet vision) for
  binding-critical use cases.** PaddleOCR's table recognizer fails on
  templates where column boundaries are spatial rather than graphical.
  A vision-LLM backend would not have this limitation but introduces
  new tradeoffs (network dependency, per-call cost, possible
  hallucination on numeric values). For specific high-value templates
  where binding accuracy matters more than throughput (e.g. AdE
  modules), an LLM backend may be worth investigating as a third
  routing option.

- **Plugin system for language- and jurisdiction-specific
  post-processors.** The Quadro RP flat-text recovery sketched in this
  report is Italy-specific (regex matching `RP\d+` codes against
  Italian-formatted monetary literals). A more general design would
  let operators ship their own post-processor for their country's tax
  forms / bank statement formats / invoice templates. The current
  routing logic has only one knob (whitelist by doctype); a plugin
  hook would let community contributions accumulate.

- **Layout-aware chunking using paddle's bbox metadata.** v1.2.4 only
  uses paddle's `markdown` and `overall_ocr_res.rec_texts` outputs;
  the `parsing_res_list` (with bounding boxes for each detected
  block) is unused. Chunking the document for downstream search by
  paddle's detected blocks rather than by character count could
  improve retrieval coherence. This is independent of the binding
  question and could be applied to any paddle-extracted document.

- **Auto-forgetting integration.** TAILOR's auto-forgetting feature
  (see `docs/AUTO_FORGETTING_TODO.md`) operates on chunk metadata.
  If paddle-routed documents ship richer per-section metadata
  (`extractor` field, `markdown` field, eventually bbox data), the
  forgetting policy could exploit it — for example, retaining
  high-binding-quality chunks longer than fallback chunks. No design
  exists for this yet.

- **Page-level routing.** Currently, routing is per-document: the
  whole file goes to paddle or to legacy. A future refinement could
  route per-page (e.g. paddle for pages with detected tables, legacy
  for prose pages), but this requires a cheap layout-classifier
  pre-pass that v1.2.4 does not implement.

## Reproducing the probes

The probe scripts were one-shot Python files written into `/tmp/` and
invoked from the TAILOR working directory. They are not part of the
shipped package.

| Stage | Script | Output | Notes |
|------:|--------|--------|-------|
| 4a | `/tmp/probe_paddle.py` | `/tmp/probe_paddle_output.txt` | Single-doc baseline on bilancio gestionale; runs the cold model download on first invocation. |
| 4a.5 | `/tmp/probe_paddle_warm.py` | `/tmp/probe_paddle_warm_output.txt` | Warm-cache timing; expects `~/.paddlex/` already populated. |
| 4a.6 | `/tmp/probe_paddle_orientation.py` | `/tmp/probe_paddle_orientation_output.txt`, `/tmp/paddle_probe_orient/run_A/`, `/tmp/paddle_probe_orient/run_B/` | A/B on `use_doc_orientation_classify` and `use_doc_unwarping`. |
| 4a.7 | `/tmp/probe_paddle_doctypes.py` | `/tmp/probe_paddle_doctypes_output.txt`, `/tmp/paddle_probe_doctypes/{estratto_conto,unico_pf,nota_integrativa}/page_*.md` | Cross-doctype binding-ratio analysis with markdown dump per page. |

To re-run any stage:

```bash
cd <tailor_root>
.venv/bin/python3 /tmp/probe_paddle_<stage>.py 2>&1 | tee /tmp/probe_paddle_<stage>_output.txt
```

The probe scripts are intentionally simple (no test harness, no
fixtures) and can be replaced/rewritten as the validation campaign
extends to new doctypes.
